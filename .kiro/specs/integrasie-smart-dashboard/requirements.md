# Requirements Document

## Introduction

IntegraSiE Smart Dashboard is an Integrated Enterprise & Smart Warehousing Platform that serves as a single source of truth for factory warehouse operations. The system eliminates manual double-entry processes (paper, chat, spreadsheets) by automating the flow from goods arrival, quality control (QC), to smart placement in the warehouse. It enforces industry regulation compliance including hazard segregation and strict cold-chain monitoring (-4°C to -20°C). The platform includes Role-Based Access Control (RBAC), enterprise audit trails, and real-time critical notification integration via WhatsApp gateway.

## Glossary

- **Platform**: The IntegraSiE Smart Dashboard application system
- **Warehouse_Operator**: A user role responsible for physical warehouse operations including goods receiving and drum arrangement
- **QC_Staff**: A user role fully responsible for laboratory testing and passing or rejecting lot numbers
- **PPIC_Team**: A user role responsible for viewing stock availability and planning manufacturing production schedules
- **Factory_Manager**: A super admin user role with full access to monitor all factory metrics, view security logs, and receive critical alerts
- **Lot_Number**: A unique identifier automatically generated by the Platform for each batch of goods received from a supplier
- **Lot_Status**: The current state of a lot, which can be "Pending QC", "Passed", "Rejected", or "Ready to Store"
- **Supplier_Intake**: The process of recording supplier delivery data into the Platform when a truck arrives
- **Smart_Slotting_Engine**: The backend algorithm that recommends optimal warehouse rack coordinates based on drum properties, hazard segregation rules, and cold-chain requirements
- **Cold_Chain_Zone**: A designated warehouse area maintained at temperatures between -4°C and -20°C for temperature-sensitive materials
- **Hazard_Segregation**: The practice of ensuring incompatible chemical substances are not stored in adjacent or nearby slots to prevent dangerous reactions
- **Audit_Trail**: A chronological record of all status changes in the Platform, capturing who performed the action, what was changed, when it occurred, the old value, and the new value
- **WhatsApp_Gateway**: A third-party API integration (such as Twilio WhatsApp API or Foneka/Waba Gateway) used to send critical alerts to relevant personnel
- **Temperature_Breach**: An event where the cold room temperature rises above the safe limit of -4°C
- **Drum**: A physical container unit of raw material stored in the warehouse
- **Rack_Coordinate**: A specific location identifier within the warehouse racking system (e.g., Block, Row, Level, Position)
- **Work_Order**: A digital production instruction created by the PPIC_Team and sent to production operators

## Requirements

### Requirement 1: Role-Based Access Control

**User Story:** As a Factory_Manager, I want the Platform to enforce role-based access control, so that each user can only access features and data relevant to their responsibilities.

#### Acceptance Criteria

1. THE Platform SHALL authenticate users via username and password before granting access to any feature
2. THE Platform SHALL assign exactly one role to each user account from the set: Warehouse_Operator, QC_Staff, PPIC_Team, Factory_Manager
3. WHEN a user with the Warehouse_Operator role logs in, THE Platform SHALL display only the Supplier Intake, Lot Queue, and Smart Slotting interfaces
4. WHEN a user with the QC_Staff role logs in, THE Platform SHALL display only the Pending QC Queue and Quality Parameter Input interfaces
5. WHEN a user with the PPIC_Team role logs in, THE Platform SHALL display only the Stock Availability Dashboard, Production Schedule, and Work Order interfaces
6. WHEN a user with the Factory_Manager role logs in, THE Platform SHALL display all interfaces including the Audit Trail, Production Efficiency Graphs, Cold Room Temperature Trends, and Alert Configuration
7. IF an authenticated user attempts to access a resource outside their role permissions, THEN THE Platform SHALL deny the request and display an access denied message indicating the user does not have the required role for the requested resource
8. IF a user submits invalid credentials, THEN THE Platform SHALL reject the login attempt, display an error message indicating authentication failure without revealing which field is incorrect, and lock the account for 15 minutes after 5 consecutive failed attempts
9. IF an authenticated user session has been inactive for more than 30 minutes, THEN THE Platform SHALL terminate the session and require the user to re-authenticate before accessing any feature
10. IF a user account is locked due to consecutive failed login attempts, THEN THE Platform SHALL notify the Factory_Manager via the Audit Trail and prevent login until the lockout period expires or the Factory_Manager manually unlocks the account

### Requirement 2: Unified Supplier Intake and Auto-Lot Generation

**User Story:** As a Warehouse_Operator, I want to input supplier delivery data once and have the system automatically generate a lot number, so that double-entry is eliminated and data integrity is maintained.

#### Acceptance Criteria

1. WHEN the Warehouse_Operator submits a Supplier_Intake form, THE Platform SHALL validate that all required fields are populated with non-empty, non-whitespace values: supplier name (maximum 100 characters), material group (selected from a predefined list), quantity (a numeric value greater than zero and no greater than 99,999), delivery date (in YYYY-MM-DD format, not a future date), and truck reference number (maximum 50 alphanumeric characters)
2. WHEN a valid Supplier_Intake form is submitted, THE Platform SHALL generate a unique Lot_Number using the format: [MaterialGroupCode]-[YYYYMMDD]-[SequentialNumber], where SequentialNumber is a zero-padded 4-digit number that resets to 0001 daily per material group
3. WHEN a Lot_Number is generated, THE Platform SHALL set the Lot_Status to "Pending QC"
4. WHEN a Lot_Number is generated, THE Platform SHALL lock the associated Supplier_Intake data to prevent modification by any user, with no override mechanism available
5. IF the Warehouse_Operator submits a Supplier_Intake form with missing or invalid required fields, THEN THE Platform SHALL reject the submission, retain the entered data in the form, and display a validation error message adjacent to each field that failed validation indicating the reason for failure
6. THE Platform SHALL display the generated Lot_Number to the Warehouse_Operator within 2 seconds of successful submission
7. IF the Warehouse_Operator submits a Supplier_Intake form with a truck reference number that matches an existing intake record submitted within the same calendar day, THEN THE Platform SHALL display a duplicate warning and require the Warehouse_Operator to explicitly confirm before proceeding with submission

### Requirement 3: QC Digital Queue and Decision Recording

**User Story:** As a QC_Staff member, I want to view a digital queue of lots pending quality control and record my test results, so that the QC process is traceable and eliminates paper-based records.

#### Acceptance Criteria

1. THE Platform SHALL display all lots with Lot_Status "Pending QC" in a chronological queue ordered by delivery date ascending (oldest first), visible to QC_Staff
2. WHEN QC_Staff selects a lot from the Pending QC queue, THE Platform SHALL display the lot details including supplier name, material group, quantity, and delivery date
3. WHEN QC_Staff submits quality parameters for a lot, THE Platform SHALL require at least one numeric or text test result entry and a decision of either "Passed" or "Rejected"
4. WHEN QC_Staff selects "Passed" for a lot, THE Platform SHALL change the Lot_Status to "Ready to Store"
5. WHEN QC_Staff selects "Rejected" for a lot, THE Platform SHALL change the Lot_Status to "Rejected" and require a rejection reason with a minimum length of 10 characters and a maximum length of 500 characters
6. WHEN a Lot_Status changes to "Ready to Store", THE Platform SHALL send a notification to the Warehouse_Operator assigned to that material group within 10 seconds of the status change
7. IF QC_Staff attempts to submit a decision without entering quality parameters, THEN THE Platform SHALL reject the submission and display a validation error indicating which parameters are missing
8. WHEN QC_Staff submits a QC decision for a lot, THE Platform SHALL lock the quality parameters and decision to prevent modification by any user after submission

### Requirement 4: Smart Slotting Recommendation Engine

**User Story:** As a Warehouse_Operator, I want the system to recommend optimal rack coordinates for storing drums, so that hazard segregation rules and cold-chain requirements are automatically enforced.

#### Acceptance Criteria

1. WHEN the Warehouse_Operator clicks "Assign Slot" for a lot with Lot_Status "Ready to Store", THE Smart_Slotting_Engine SHALL analyze the drum's material group, hazard classification, and temperature sensitivity, and recommend between 1 and 5 valid Rack_Coordinates within 3 seconds
2. WHILE a Drum is classified as temperature-sensitive based on its material group designation, THE Smart_Slotting_Engine SHALL recommend only Rack_Coordinates within the Cold_Chain_Zone maintained between -4°C and -20°C
3. WHILE a Drum contains hazardous chemicals, THE Smart_Slotting_Engine SHALL recommend only Rack_Coordinates where all directly adjacent slots (same row: left and right positions; same position: above and below levels) are either empty or contain compatible substances according to the hazard segregation matrix
4. WHEN the Smart_Slotting_Engine cannot find a valid Rack_Coordinate that satisfies all constraints, THE Platform SHALL display a "No Available Slot" warning to the Warehouse_Operator and send a notification to the Factory_Manager via the WhatsApp_Gateway within 30 seconds
5. WHEN the Warehouse_Operator confirms placement at a recommended Rack_Coordinate, THE Platform SHALL update the slot status to "Occupied", record the Lot_Number at that coordinate, and create an Audit_Trail record of the placement
6. IF the Warehouse_Operator attempts to place a Drum at a non-recommended Rack_Coordinate, THEN THE Platform SHALL display a compliance warning stating the violated constraint, and require the Warehouse_Operator to enter an override justification of at least 10 characters before allowing placement
7. WHEN the Warehouse_Operator confirms an override placement at a non-recommended Rack_Coordinate, THE Platform SHALL record the override justification in the Audit_Trail and send a notification to the Factory_Manager via the WhatsApp_Gateway

### Requirement 5: Cold-Chain Temperature Monitoring

**User Story:** As a Factory_Manager, I want the system to continuously monitor cold room temperatures and alert me immediately when a breach occurs, so that temperature-sensitive materials are protected.

#### Acceptance Criteria

1. THE Platform SHALL read temperature sensor data from all Cold_Chain_Zone areas at intervals no greater than 60 seconds
2. THE Platform SHALL display temperature readings for each Cold_Chain_Zone on the Factory_Manager dashboard, updated within 5 seconds of each sensor read
3. WHEN a temperature reading in any Cold_Chain_Zone exceeds -4°C, THE Platform SHALL classify the event as a Temperature_Breach
4. WHEN a Temperature_Breach occurs, THE Platform SHALL send a WhatsApp alert to the Factory_Manager and Warehouse Maintenance Team within 30 seconds of detection
5. WHEN a Temperature_Breach occurs, THE Platform SHALL include in the alert message: the affected zone identifier, the current temperature reading, and the safe temperature limit
6. WHILE a Temperature_Breach is active, THE Platform SHALL display a visual alarm indicator on the Factory_Manager dashboard and the Warehouse_Operator interface until the affected Cold_Chain_Zone records 3 consecutive readings at or below -4°C, at which point the breach status SHALL be resolved and the alarm indicator removed
7. IF a Cold_Chain_Zone temperature sensor fails to report data for more than 2 consecutive polling intervals (120 seconds), THEN THE Platform SHALL classify the event as a sensor communication failure and send a WhatsApp alert to the Factory_Manager and Warehouse Maintenance Team indicating the affected zone and the timestamp of the last successful reading

### Requirement 6: Enterprise Audit Trail

**User Story:** As a Factory_Manager, I want every status change in the system to be automatically recorded with full context, so that factory operations are transparent and traceable.

#### Acceptance Criteria

1. WHEN any Lot_Status changes, THE Platform SHALL create an Audit_Trail record containing: the user identifier, the action performed, the timestamp in ISO 8601 UTC format with seconds precision, the previous Lot_Status value, and the new Lot_Status value
2. WHEN any Drum location changes, THE Platform SHALL create an Audit_Trail record containing: the user identifier, the action performed, the timestamp in ISO 8601 UTC format with seconds precision, the previous Rack_Coordinate (or "None" if this is the initial placement), and the new Rack_Coordinate
3. THE Platform SHALL store Audit_Trail records in an append-only manner that prevents modification or deletion by any user including the Factory_Manager
4. WHEN the Factory_Manager accesses the Audit Trail interface, THE Platform SHALL display records in reverse chronological order with pagination of no more than 50 records per page, and provide filtering options by date range, user, action type, and lot number
5. THE Platform SHALL retain all Audit_Trail records for a minimum of 5 years from the date of creation
6. IF the Platform fails to create an Audit_Trail record for a status change, THEN THE Platform SHALL block the triggering operation, revert the status change, and display an error message indicating that the action cannot be completed due to an audit recording failure

### Requirement 7: Critical Notification via WhatsApp Gateway

**User Story:** As a Factory_Manager, I want critical operational alerts delivered to my WhatsApp, so that I can respond immediately to anomalies without needing to be logged into the Platform.

#### Acceptance Criteria

1. THE Platform SHALL integrate with a WhatsApp_Gateway API to deliver critical alert messages for the following event types: Temperature_Breach, Smart_Slotting_Engine failure (no available slot), and WhatsApp_Gateway delivery failure after all retry attempts are exhausted
2. WHEN a critical event occurs, THE Platform SHALL format the alert message with: severity level, event description, affected area, current reading, safe threshold, and recommended action, with the total message length not exceeding 1000 characters
3. WHEN a critical alert is triggered, THE Platform SHALL deliver the WhatsApp message to all designated recipients for that alert category (up to a maximum of 20 recipients per category) within 30 seconds of event detection
4. IF the WhatsApp_Gateway returns a delivery failure, THEN THE Platform SHALL retry delivery up to 3 times with 10-second intervals, log each failed attempt in the Audit_Trail, and IF all 3 retries fail, THEN THE Platform SHALL log a final failure record in the Audit_Trail and display a persistent unacknowledged alert banner on the Factory_Manager dashboard upon next login
5. THE Platform SHALL allow the Factory_Manager to configure which user phone numbers receive critical alerts for each alert category (Temperature_Breach, Smart_Slotting_Engine failure), requiring each phone number to be in valid international format (E.164) and requiring at least one recipient per alert category
6. IF the Factory_Manager attempts to save an alert configuration with an invalid phone number format, THEN THE Platform SHALL reject the configuration and indicate which phone number entries are invalid

### Requirement 8: PPIC Stock Visibility and Production Planning

**User Story:** As a PPIC_Team member, I want to view real-time stock of QC-passed raw materials and create production schedules, so that manufacturing planning is based on accurate, live data.

#### Acceptance Criteria

1. THE Platform SHALL display a dashboard showing all lots with Lot_Status "Passed" or "Ready to Store", including material group, quantity, warehouse location (Rack_Coordinate), and Lot_Number, with data refreshed at intervals no greater than 30 seconds
2. WHEN a Lot_Status changes to "Ready to Store", THE Platform SHALL update the PPIC dashboard stock count within 5 seconds
3. WHEN the PPIC_Team creates a production schedule, THE Platform SHALL require the following fields: target product, list of required raw material lots with quantities, and planned production date
4. WHEN the PPIC_Team submits a production schedule, THE Platform SHALL validate that for each referenced raw material, the requested quantity does not exceed the available quantity of lots with Lot_Status "Ready to Store"
5. WHEN the PPIC_Team issues a Work_Order, THE Platform SHALL send a notification to the designated production operators within 30 seconds, containing: the Work_Order identifier, target product, list of required materials with lot numbers and quantities, and planned production date
6. IF the PPIC_Team attempts to schedule production using lots that do not have Lot_Status "Ready to Store", THEN THE Platform SHALL reject the schedule and display the current Lot_Status of each referenced lot that is not "Ready to Store"
7. WHEN a Work_Order is issued, THE Platform SHALL reserve the referenced lot quantities so that other production schedules cannot allocate the same stock
8. IF the PPIC_Team attempts to schedule production referencing lots whose unreserved quantity is insufficient for the requested amount, THEN THE Platform SHALL reject the schedule and display the available unreserved quantity for each insufficient lot
